# Franchise Tag Module Review

## Executive Summary

The current Franchise Tag module is driven by `site/ccc/tag_tracking.json`, generated by `pipelines/etl/scripts/build_tag_tracking.py`, and rendered in the browser by `site/ccc/ccc.js`.

The older/original calculation logic was recoverable from repo history, primarily the mid-February 2026 implementation introduced in commit `5be049a` and the immediately preceding development history such as `da4b09f` and `6ac2876`.

The recovered result is:

- The modern module does **not** use the legacy rulebook's simple `Top 5` franchise tag average.
- It uses a **tiered positional model** with fixed rank bands and fixed AAV averaging bands by position.
- The current implementation has grown beyond the February baseline:
  - it now includes expiring rookies,
  - it allows a fallback salary for unranked expiring players,
  - it excludes some players because of extension lineage,
  - and the browser can recalculate a higher displayed tag number than the stored ETL output.

The biggest review findings are:

1. `site/ccc/tag_tracking.json` is currently stale for 2026 usage. It was generated on `2026-03-07` but still carries `season=2025`, `tracking_for_season=2025`, and `exclude_tagged_season=2024`.
2. The ETL and UI do not use exactly the same bump-floor logic. This can change the rendered tag number.
3. The code comments and notes still describe parts of the old process inaccurately.
4. The current module is a **Franchise Tag-only** implementation. The legacy **Transition Tag** process was not carried forward.

## Current Franchise Tag Process

### Current source data

The current flow is:

1. Pull expiring rostered players from `rosters_current` where:
   - `contract_year = 1`
   - roster status is `ROSTER` or `INJURED_RESERVE`
2. Normalize positions into tag groups:
   - `QB`, `RB`, `WR`, `TE`, `DL`, `LB`, `DB`, `PK`
3. Build current-season scoring ranks from `player_weeklyscoringresults`
   - total points across weeks `1-last_regular_week`
   - not `player_pointssummary.pos_rank`, despite the file header comment
4. Build the salary source pool from week 1 contract snapshots
   - primarily `rosters_weekly`
   - adjusted by `contract_history_snapshots` when available
5. Compute tier base bids from the week 1 AAV pool
6. Apply player-specific floors/exclusions
7. Write `meta.calc_breakdown` plus one row per tracked candidate to `tag_tracking.json`
8. Render:
   - eligible player table
   - ineligible list
   - cost calc breakdown
   - summary view
   - finalized submissions

### Current formula

For most positions, the ETL computes:

`tier_base_bid = round_up_to_1000(average(week_1_AAVs_of_source_players_in_band))`

Then, for the tracked player:

`bump_base = max(prior_aav_week1, current_salary)`

`bump_floor = round_up_to_1000(bump_base * 1.10)`

`tag_bid = max(tier_base_bid, bump_floor)`

For `PK/PN`:

`base_bid = max(1000, prior_aav_week1 + 1000)`

`tag_bid = max(base_bid, bump_floor)`

For unranked expiring players with no tier rule match:

`fallback_base = max(1000, max(prior_aav_week1, current_salary))`

`tag_bid = max(fallback_base, bump_floor)`

## Historical / Original Process

### Legacy rulebook process

The older league documents used a simpler process than the modern tier engine:

- 2014 bylaws: franchise tag minimum salary had to meet the **top 5 average** at the position and exceed the player's current contract.
- 2018 rules: franchise tag opening bid was **higher of**
  - prior season AAV plus 1,000
  - average of the top 5 players at the position
- 2018 rules also had a distinct **Transition Tag**
  - higher of prior season AAV plus 1,000 or average of the top 10 players at the position

That means the historical league process and the current module are not the same design.

### Recovered original module logic from mid-February 2026

The original recoverable module logic is the February 16, 2026 baseline in repo history:

- commit `5be049a`
- supported by earlier dev history such as `da4b09f`

That recovered baseline already used the tiered model above, not the old top-5-only rulebook structure.

Recovered February baseline formula:

- use the same positional tier bands now stored in `TAG_RULES`
- for non-kickers:
  - `tag_bid = tier_base_bid`
  - if `prior_aav_week1 >= tier_base_bid`, then override with `round_up_to_1000(prior_aav_week1 * 1.10)`
- for `PK/PN`:
  - `tag_bid = prior_aav_week1 + 1000`
- no fallback for unranked players
- no extension-lineage exclusion
- notes still described the pool as `non-rookie expiring contracts`

In other words, the recoverable "original module" was:

- already tiered
- narrower than the current implementation
- easier to reason about

## Current vs Historical Comparison

### What stayed the same

- the tier table itself
- the use of week 1 AAV snapshots as the cost source pool
- the use of current-season scoring rank to place a player into a tier
- special `PK/PN` handling
- `calc_breakdown` source-player storage

### What changed from the recovered February baseline

1. **Bump logic widened**

Recovered baseline:

- bump only when `prior_aav_week1 >= tier_base_bid`
- bump source was effectively `prior_aav_week1`

Current ETL:

- bump source is `max(prior_aav_week1, current_salary)`
- called `10% salary floor`

Current UI:

- recomputes bump source as `max(prior_aav, prior_salary, current_salary, parsed contract AAV, max year salary)`
- this is wider than ETL

2. **Fallback path was added**

Recovered baseline:

- players without a rank/tier match were not tag-eligible

Current:

- unranked expiring players can still receive a fallback tag salary

Examples in current file:

- Joe Mixon
- Will Levis

3. **Rookie handling changed**

Recovered baseline notes still said non-rookie expiring contracts.

Current notes explicitly say:

- expiring contracts including rookies

4. **Extension-lineage exclusion was added**

Current logic blocks tag eligibility if a different franchise is treated as the last extension holder.

This did not exist in the February baseline.

5. **Tracking season behavior drifted**

The ETL contains rollover logic intended to move tagging into the upcoming season, but the current shipped `tag_tracking.json` still shows `tracking_for_season=2025` even though it was generated on `2026-03-07`.

## Gaps / Issues / Suspected Problems

1. **Current file is stale for 2026**
   - `generated_at=2026-03-07`
   - `season=2025`
   - `tracking_for_season=2025`
   - `exclude_tagged_season=2024`

2. **ETL and UI disagree**
   - `build_tag_tracking.py` and `ccc.js#getTagBidDetails()` do not use the same bump inputs

3. **File header comments are outdated**
   - the script header still claims ranking source is `player_pointssummary.pos_rank`
   - the actual implementation ranks from `player_weeklyscoringresults`

4. **Rulebook mismatch**
   - the legacy rulebook describes top-5 / top-10 tag rules
   - the current module is a custom tier engine
   - transition tags are absent from the current module

5. **Kyler exclusion is UI-hardcoded**
   - the exclusion exists in `ccc.js`
   - it is not explained in the ETL data model itself

6. **PK calc storage is incomplete as a summary**
   - `calc_breakdown` stores PK source players but base bid stays `0`
   - the actual bid is per-player

7. **The current report is still harder to validate than it should be**
   - the source players exist in the payload
   - but the owner-readable explanation is still buried under tabs/details

## Are the tiers/calculations still correct?

### Tier bands

The tier bands are internally consistent between:

- `build_tag_tracking.py`
- `ccc.js`
- `tag_tracking.json`

So the tier table itself appears intact.

### Calculation correctness

The answer is mixed:

- the stored ETL tier bases appear internally consistent with the current `calc_breakdown`
- the live rendered cost can differ from stored ETL because the browser recomputes the salary floor
- the file currently shipped to the site appears to be for the wrong season context

So the calculations are **not fully trustworthy end-to-end** until:

1. the season rollover/output issue is fixed
2. ETL and UI share one exact bump-floor implementation

## Recommendations

1. Make ETL the single source of truth for final tag bid.
2. Remove the separate browser-side recalculation or make it call the exact same formula inputs.
3. Rebuild `tag_tracking.json` for the correct 2026 tracking season.
4. Explicitly version the rule set:
   - `legacy_rulebook_tag_process`
   - `modern_tiered_tag_process`
5. Surface exception flags directly in the report:
   - prior-tag exclusion
   - extension-lineage exclusion
   - fallback salary used
   - hard-coded exclusion
6. Decide whether expiring rookies are truly intended in the modern rule set and document it.
7. Add a first-class report page outside the CCC action UI if long-term referenceability matters.

## Reconstructed Answer to the Key Questions

1. **How is the franchise tag currently calculated?**
   - by current-season scoring tier, then by week 1 AAV average of the position's salary source band, then by a 10% salary floor override, with special PK and fallback handling

2. **What are the current tag tiers?**
   - the tier table listed above in the Current tier structure section

3. **Which players make up each tag cost calculation?**
   - exported in `docs/franchise_tag_calculation_breakdown.csv`; current examples are listed above and the full source pools are embedded in `meta.calc_breakdown`

4. **What was the original calculation logic?**
   - recovered February baseline: same tier engine, narrower prior-AAV-only bump rule, no fallback, no extension-lineage exclusion

5. **Did the current implementation lose any detail that used to exist?**
   - yes: the old simpler explanation was easier to reason about; more importantly, the current implementation lost one-source-of-truth behavior because UI and ETL diverge

6. **Are the tiers/calculations still correct?**
   - tiers appear consistent; end-to-end calculation output is not fully reliable because of season staleness and ETL/UI mismatch

7. **What should be surfaced in the report/module so the logic is easily visible?**
   - tier tables, source-player pools, base cost, floor override, final cost, and exception flags in one owner-readable report
