How to use this page

Back to top

This is a working rulebook browser. It is meant to be read in logical order, searched, and cross-referenced against the governed UPS_V2 docs. Anything marked skeleton is intentionally incomplete. Anything marked pending review is a real working draft, not final truth.

Search criteria

  • Rule family: trade, contract, auction, roster, add, drop, enforcement, site, scoring.
  • Source tier: governed, V1 reference, pending review, skeleton.
  • Status: settled, partial, open, skeleton.
  • Identifier: ambiguity ID, directive ID, SSA row, or source file name.
  • Plain language: player kicked off, taxi-on-trade, 48 hours, 3 days, noon open, 24-hour timer.

Reading rule

  • V1 reference material is evidence only.
  • Governed UPS_V2 docs are the implementation target.
  • If the page says pending review, it means the rule is visible but not closed.
  • If the page says skeleton, it is a placeholder waiting for later promotion.

Authority and precedence

Back to top

UPS_V2 should resolve rule conflicts by source tier, not by convenience. Commissioner directives and explicit governing docs outrank precedent. Seasonal MFL metadata outranks narrative silence when the rulebook is silent. V1 forum, Discord, and message-board evidence is support material unless promoted.

Precedence ladder

  1. Explicit commissioner directive or governed UPS_V2 approval.
  2. Live season MFL metadata when the rulebook is silent.
  3. Historical precedent from forum, Slack, Discord, or message boards.
  4. Codex-derived summaries and derived notes.

Practical rule

  • Do not hardcode commissioner interpretation as settled law.
  • Log drift when live site settings and written rules disagree.
  • Promote only what has been reviewed and approved.
Example: source conflict
If a forum thread says one thing and the current season metadata says another, UPS_V2 should treat the metadata as operational truth if the rulebook is silent. If the commissioner later posts an explicit directive, that directive becomes the governing decision and the old conflict is preserved in the ambiguity log.

Metadata and source of truth

Back to top

The canonical metadata register merges league settings, divisions, starters, and event bounds by season. Raw season metadata is authoritative for the season it describes. Rendered summaries are convenience only.

Metadata fields treated as operational truth

  • league_id and season
  • salary cap amount and cap mode
  • roster size, taxi, IR, and starter counts
  • division and conference structure
  • start and end weeks, event windows, and roster lock windows

Convenience only

  • human-readable labels for divisions and conferences
  • derived summary counts
  • generated calendar cards
  • readable validation warnings
Example: metadata overrides narrative silence
If a rulebook page does not specify a starter count for a season but the season metadata does, UPS_V2 should use the metadata in the live validator and keep the mismatch visible in the site-settings alignment register.

League calendar and event windows

Back to top

The current calendar logic is season-configured, not hardcoded. The key pattern is the same each year even when dates move: offseason release and preparation, rookie and contract activity, auction period, kickoff lock, Thanksgiving trade deadline, and postseason closeout.

Anchor events

  • Season start and offseason rollover.
  • Rookie draft and rookie contract deadlines.
  • Free-agent auction target period, usually the last weekend of July but flexible.
  • Final cut-down day, normally within 48 hours before auction start.
  • Trade deadline, tied to the Thanksgiving kickoff window.

Behavior rule

  • Timing should be operationally fluid unless it breaks a governed rule.
  • Season event timing is not a violation by itself.
  • Every exception should still be logged.
Example: auction timing flexibility
If commissioner availability pushes the auction away from the last weekend of July, UPS_V2 should treat that as a planned operating shift, not as a rule violation, as long as the season metadata and commissioner directive are aligned.

Roster architecture

Back to top

The roster model should distinguish in-season roster, taxi squad, injured reserve, starter requirements, and roster lock periods. The exact counts vary by season, so the page should always point to season metadata first and rule text second.

Core roster concepts

  • Active roster and bench.
  • Taxi squad and taxi eligibility.
  • Injured reserve and cap treatment.
  • Start/sit legality and weekly lineup validation.

Operational rule

  • Roster legality should be checked before publishable actions are finalized.
  • Taxi-on-trade behavior is not optional once a trade is accepted.
  • Roster lock behavior may differ before the auction versus in-season.
Example: taxi player traded to another team
If a traded player is on taxi, the player should remain on taxi after the trade and be auto-demoted back onto taxi if the incoming team accepts the trade while the player is still taxi-eligible.

Contracts and contract mechanics

Back to top

This is the most important section for examples. Contracts should always be read as a state machine, not just a dollar amount. When the rulebook is ambiguous, show the raw rule, the governed interpretation, and the example outcome side by side.

Contract family First-pass rule Example Status
Rookie contract Draft class contracts are season-configured; the historical move to 3-year rookie deals belongs in the rule timeline. A rookie signs for 3 years, then can later become extension eligible or option eligible depending on class and season rules. partial
Veteran contract Veteran deals should track term, cap hit, guarantees, and current-season salary separately. A 2-year veteran contract can be front-loaded, back-loaded, or loaded if the governing contract model allows it. partial
Extension Extensions add term or change structure through governed approval, not by free-form text. Player A has one year left and gets extended for two additional years under the extension rules. partial
Restructure Restructure changes salary profile without pretending the original contract never existed. Player B's cap hit is reduced in-year, with the change logged as a restructure event, not a new contract. partial
MYM MYM is a governed action with its own eligibility and timing. A player is eligible for MYM only if the season metadata and contract state say so. partial
Option year Option-year logic is season-specific and should not be inferred from a single old example. A 2025 draft class player may have an option path that later becomes live in the governance model. skeleton
Tags Tags need a separate decision path because historical sources show they existed and later drifted away from the current calendar. Franchise or transition tag handling is either promoted into the rulebook or explicitly parked as inactive. open
Example: rookie extension versus veteran extension
Rookie and veteran extensions may use the same approval workflow, but they are not the same rule. A rookie extension is governed by the rookie class path and the remaining rookie-year context. A veteran extension is governed by the veteran contract path and the active term that is already on the ledger.
Example: restructure without erasing history
If a player's cap hit is rebalanced, the original contract record stays intact and a new restructure event is written on top of it. That way the ledger shows both the original deal and the later adjustment.
Example: MYM eligibility
A team should not be allowed to apply an MYM action just because it is convenient. The player must be eligible under the season's metadata, and the action must be logged as its own contract event.

Auction and free agency

Back to top

The auction model should be readable in one pass: when it opens, how nominations reset, when it stops, and what happens if the nomination window is missed. The current governed path is noon open, midnight reset, two nominations per cycle, and nomination-complete stop logic.

Auction rules

  • Normal target period: last weekend of July.
  • Open time: noon Eastern.
  • Reset: midnight Eastern with a fresh nomination allowance.
  • Stop: nomination-complete ends the cycle.

Pending review

  • Exact missed-nomination penalty ladder.
  • Any special exceptions to the stop logic.
  • How long late reopenings stay open.
Example: first day of auction
A franchise gets two nominations on the opening day, the queue resets at midnight, and the system keeps accepting nominations until the nomination-complete condition is reached.
Example: missed nomination
If a franchise misses a nomination opportunity, the system should not invent the penalty ladder. It should display the current governed ladder if it exists, or mark the penalty path open for commissioner or rulebook review.

Adds and waivers

Back to top

Adds should be read as availability plus compliance plus schedule. The current live behavior in the governed notes is BBID_FCFS with blind bids Thursday through Sunday at 9 AM and FCFS after the Sunday waiver run until kickoff.

Add window

  • Start is derived from the first configured waiver run after auction completion.
  • End is derived from season metadata and the last regular season week.
  • Timing can shift season to season if the metadata and live settings are aligned.

Validation

  • Roster and cap legality must be checked before the add is final.
  • Waiver mode should be cross-checked against live MFL site settings.
  • Keep the site-settings alignment register current.
Example: post-auction add window
If the auction ends on Saturday night and the first waiver run is Wednesday morning, the add window should start at the first configured waiver run after auction completion, not at a hardcoded calendar day.

Drops and roster locks

Back to top

Drops need a hard legality gate because they can create irreparable state changes if they are allowed too late. The governing model should prevent drops when the player has already kicked off, when the auction lock is active, or when a publish or commissioner hold exists.

Drop blockers

  • Player not rostered by the team.
  • Player has already kicked off.
  • Pre-auction roster lock is active.
  • Auction is active and the roster lock is in force.
  • Workflow, publish, or commissioner hold is present.

Timing rule

  • Pre-auction drop lock is season-configured.
  • The normal target is within 48 hours of auction start.
  • The exact setting should stay configurable.
Example: player already kicked off
If a player has already played in the current scoring week, the drop request should hard block. The system should not guess at a retroactive correction unless a commissioner-level recovery workflow exists.

Trades and negotiation

Back to top

Trades are the most operationally delicate part of the rule system. The current UPS_V2 posture is: negotiate in Discord DMs, record state in UPS_V2, accept on the website, and keep live cross-checks visible. Trade window timing is season-configured and closes at the Thanksgiving kickoff deadline.

Governed trade assumptions

  • Trade negotiation can begin at the new league year.
  • Trade offers can be discussed in Discord, but acceptance belongs on the website.
  • If the player has already kicked off, prefer deferred next-week processing if the source system allows it.
  • Release 1 is self-serve two-team only.
  • Taxi-on-trade returns the player to taxi when valid.

Open trade questions

  • Exact response SLA by season.
  • Whether kickoffs hard block acceptance or defer it when source support exists.
  • What happens when a trade leaves a team temporarily over cap.
  • When multi-team trades graduate from commissioner-managed exceptions.
Example: kicked-off player in a trade
If one side wants to accept a trade after a player in the deal has already kicked off, UPS_V2 should not pretend the original moment still exists. If the source system supports it, the action should move to next week. If not, store the intent locally and require a later acceptance window.
Example: taxi-on-trade
If a traded player is on taxi and the trade is accepted, the player should remain on taxi in the recipient's system view and the taxi salary should stay excluded from the trade salary limit.
Example: trade salary arithmetic
If a team trades away two active players with current-season salaries of 8K and 6K, the trade salary limit is based on half of the outgoing active or IR salary pool, excluding taxi. That means the salary component is derived from the outgoing pool, not negotiated ad hoc.
Example: Discord negotiation versus final acceptance
The Discord thread can say "sounds good" or "send it," but the governed state only changes when UPS_V2 records the acceptance event on the website or in a governed publish workflow.

Penalties and enforcement

Back to top

Enforcement should separate warnings from hard blocks. Not every mismatch is the same. Some items should warn only, some should soft block, some should hard block, and some should be commissioner overrideable.

Enforcement ladder

  • Warn only: user can proceed, but the state is logged.
  • Soft block: user can recover with corrections.
  • Hard block: action cannot proceed.
  • Commissioner override allowed: break-glass path with audit.
  • Commissioner override forbidden: no exception path.

Open penalty items

  • Exact auction missed-nomination ladder.
  • Over-cap handling after trade or during auction.
  • Any remaining 24-hour wording cleanup from V1.
Example: warning versus hard block
A roster that is one player short of a preferred depth target can warn only. A trade that violates the salary cap or a drop that happens after kickoff should hard block unless a commissioner's governed override exists.

Site behavior and workflow controls

Back to top

Site behavior is not just UI. It includes actor permissions, publish objects, verification types, site-settings cross-checks, and the way outside tools like Discord are treated.

Workflow rules

  • Discord DM interaction is transport only.
  • Acceptance belongs in UPS_V2 and should be auditable.
  • Site settings must be cross-checked against live source system behavior.
  • Publisher state should remain explicit and reviewable.

Publish object model

  • publish_unit_type
  • publish_scope
  • approval_class
  • verification_type
Example: DM versus governed state
A Discord DM can say that a trade is accepted, but UPS_V2 should still wait for the governed acceptance event. If the DM says one thing and the site says another, the site wins until the mismatch is resolved.

Examples appendix

Back to top

This appendix exists so the hard parts are concrete. The examples are illustrative, not a promise that every edge case is already closed.

Contract examples

  • Rookie contract: a draft-class player signs a 3-year deal and later becomes extension eligible.
  • Extension: a one-year-left veteran gets two more years through governed approval.
  • Restructure: cap hit changes, original contract record remains visible.
  • MYM: only if the season metadata says the player is eligible.

Trade and auction examples

  • Trade with a kicked-off player: defer if possible, otherwise store the intent and wait.
  • Taxi-on-trade: incoming taxi player stays taxi.
  • Auction: noon open, midnight reset, two nominations per cycle.
  • Drop: hard block after kickoff or during an active auction lock.
Detailed example: rookie contract and extension
A rookie drafted in a 3-year contract era enters the ledger with term, cap hit, and eligibility state. If the player later becomes extension eligible, the extension is recorded as a new governed action on top of the original deal. The old contract does not disappear.
Detailed example: trade salary and cap treatment
If a trade includes salary, UPS_V2 should derive the salary limit from the outgoing active-roster or IR current-season salary pool, excluding taxi. The trade can then be validated against cap and roster constraints before it becomes final.
Detailed example: auction nomination behavior
If a franchise opens the day with two nominations and then misses the next cycle, the page should show the active nomination state, the penalty policy state, and any open ambiguity instead of hiding the gap behind vague verbiage.
