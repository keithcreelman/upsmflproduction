League basics

Back to top

This is the standard opening section you would expect in a fantasy rulebook: league identity, team structure, divisions, season shape, and the broad shape of play. It should be readable before any player-specific rule.

League structure

  • 12-team dynasty format.
  • 2 conferences and 2 divisions per conference in the long-standing structure.
  • Season calendar includes offseason, auction, regular season, and playoffs.
  • Commissioner and co-commissioner roles exist for governed control.

League behavior

  • Owners negotiate, roster, and manage contracts inside the site.
  • Discord and message boards can carry discussion, but the site owns the governing state.
  • Calendar shifts are allowed if they are intentionally governed and visible.
Example: season timing shift
If the auction moves a week because the league needs it to, that is not automatically a rule violation. The move should be reflected as a governed season choice and the site should show the new timing clearly.

Roster and lineup rules

Back to top

This section should tell an owner exactly what is legal on a roster, what is legal in a lineup, and what gets blocked. It should read like the rules of play, not like a data model.

Roster zones

  • Active roster.
  • Taxi squad.
  • Injured reserve.
  • Waiver / free-agent pool.

Lineup behavior

  • A player must be eligible to be started.
  • The site should show what is legal, not just what exists in the database.
  • Weekly roster legality should be visible before publishable actions finalize.
Example: drop a player after kickoff
If a player has already kicked off in the current scoring week, a drop request should hard block unless a specific commissioner recovery rule exists. The user should see the block reason immediately.
Example: taxi player promoted by mistake
If a player belongs on taxi but is accidentally sitting on the active roster, the roster workspace should show the mismatch and hand the owner into the correct roster action, not just tell them something is wrong.

Transactions

Back to top

Transactions should cover adds, drops, waivers, roster locks, and the timing rules that govern when a transaction is legal. The page should explain the normal behavior first and the edge cases second.

Adds and waivers

  • Adds are tied to waiver or FCFS legality.
  • Waiver and add timing should be season-configured.
  • Post-auction adds should not open until the league is past the auction lock point.

Drops and locks

  • Pre-auction roster lock applies before the auction window.
  • Dropping after kickoff or during an active lock should block.
  • Commissioner or workflow holds should be visible, not hidden.
Example: pre-auction cut-down
If final cut-down is normally within 48 hours of auction start, the site should display that as a league operating target. It should not pretend every season must use the exact same clock time forever.

Auction and free agency

Back to top

This section covers the player acquisition window everyone expects to find in a fantasy rulebook: when the auction opens, how nominations work, how long bids stay live, and what happens if a franchise misses a nomination.

Core auction behavior

  • Typical target period: last weekend of July.
  • Open time: noon Eastern.
  • Reset: midnight Eastern.
  • Nomination-complete can end a franchise’s nomination duty.

Open / pending review

  • Exact missed-nomination penalty ladder.
  • Any exception to the stop / supplemental nomination logic.
  • Whether auction-related cap adjustments need a special workflow.
Example: nomination-complete
If a franchise has enough valid players on the board to satisfy roster completion, it may stop nominating. If it later loses a required leading bid, the rulebook should say whether a supplemental nomination is allowed to restore completion.
Example: missed nomination
Missed nominations should show the current governing penalty path. If the ladder is still open, the page should say that plainly instead of pretending the rule is settled.

Contracts

Back to top

This is the section that needs the most examples because contract wording is where fantasy rulebooks tend to get muddy. The right model is a lifecycle: a contract exists, changes state, and may be extended, restructured, tagged, optioned, or adjusted by MYM.

Contract family What it means Example Status
Rookie contract Draft-class deal with its own term and eligibility path. A rookie signs a 3-year deal and later becomes extension-eligible. partial
Veteran contract Non-rookie deal with term, cap hit, and salary behavior. A veteran has a two-year deal that can later be extended or restructured. partial
Extension Adds or reshapes term under governed eligibility rules. One year left becomes three years after extension approval. partial
Restructure Changes salary profile without erasing the original contract. Cap hit drops this year and the restructure event is logged on top of the old deal. partial
Tag Special franchise/transition style decision with its own governance question. Tag stays active only if the current rule system says it is still live. open
MYM Distinct governed contract action with its own eligibility. MYM applies only when the season and contract state permit it. partial
Option year Special rookie-path contract year with explicit salary logic. First-round rookie option is its own governed state, not a hidden price tweak. partial
Example: extension versus restructure
An extension adds new term. A restructure changes how the current money hits the cap. They may happen around the same kind of player, but they are not the same action and should not be modeled as the same rule.
Example: option year
If an option year is available for a first-round rookie, the page should explain that it is a distinct action with a distinct salary effect. If the defensive version is still under review, say so plainly and mark it open.
Example: MYM
MYM should not be written as vague lore. The page should say who can use it, when it is legal, and what state change it creates when executed.

Trades

Back to top

Trades deserve a full section because they are the place where most leagues get confusing: window timing, kickoff lock, DM chatter, salary movement, taxi behavior, and multi-team exceptions all collide. The rulebook should make those collisions obvious.

Core trade rules

  • Trade window begins at the new league year.
  • Trade window stays open through offseason, auction, and regular season until Thanksgiving kickoff.
  • Trade acceptance after kickoff should be blocked or deferred according to the governed rule.
  • Taxi-on-trade returns the player to taxi when valid.

Communication rules

  • Discord DMs are transport and lightweight response UI.
  • Website acceptance is canonical.
  • Final trade approval should not live only in chat.
  • Multi-team trades remain commissioner-managed until explicitly promoted.
Example: kicked-off player in a trade
If one player in the deal has already kicked off, the site should not pretend the current scoring week still exists for that action. If the source system allows deferral, move the trade to the next eligible week. If it does not, store the intent and wait for the governed acceptance path.
Example: taxi-on-trade
If a traded player is on taxi and remains taxi-eligible after the move, the acquiring team should receive that player back on taxi rather than as a normal active-roster player.
Example: trade salary
If a trade includes salary movement, the salary limit should be derived from the outgoing active-roster or IR current-season salary pool, with taxi excluded.

Penalties and enforcement

Back to top

A fantasy rulebook needs an enforcement section that says what warns, what blocks, what can be overridden, and what cannot be overridden. It should be blunt enough that an owner can tell what happened without guessing.

Enforcement ladder

  • Warn only: user can continue, but the issue is logged.
  • Soft block: user can recover by fixing the issue.
  • Hard block: action cannot proceed.
  • Commissioner override allowed: break-glass path with audit.
  • Commissioner override forbidden: no exception path.

Open items

  • Missed auction nomination ladder.
  • Over-cap treatment after trade or auction.
  • Penalty wording that still relies on old placeholder text.
Example: block versus warning
A lineup note that says a team is running thin at one position can warn only. A trade that violates a hard salary rule should block. The page should not use the same color or language for both.

Site surfaces and navigation

Back to top

This is the part that should make the site feel sane. The user should understand where to go for planning, player actions, contract actions, trades, reports, and commissioner tools without having to decode old shell labels.

Owned surfaces

  • Team Operations: franchise planning and cap context.
  • Roster Operations: player-action launcher.
  • Contract Actions: extensions, restructures, tags, MYM, options, auction finalization.
  • Trade Negotiation: negotiation, acceptance, response tracking.

Supporting surfaces

  • Reporting: standings, history, analytics, summaries.
  • Commissioner tools: approvals, overrides, audit review.
  • Rulebook / Help: this document and related guides.
Example: where a user should go
If the user wants to extend a player, the natural path is not a generic settings page. It is roster context into contract actions. If they want to compare two teams’ cap situations, that starts in Team Operations or Reporting.

Examples appendix

Back to top

The goal of the examples appendix is to remove ambiguity before the user has to ask for it. If a rule can be misunderstood, show a concrete case.

Contract examples

  • Rookie contract becomes extension eligible after its governed term.
  • Extension adds new term without erasing the original contract record.
  • Restructure changes cap flow without pretending a new player was signed.
  • MYM is only legal if the player’s eligibility state allows it.

Trade and auction examples

  • Trade with a kicked-off player is deferred if the source system supports it.
  • Taxi player traded in stays taxi when valid.
  • Auction nominations reset at midnight and stop at nomination-complete.
  • Drop after kickoff is blocked.
Detailed example: contract lifecycle
A player is drafted, signs a rookie deal, then later gets extended. The page should show the player’s original contract, the extension event, and the current active state. The old state should still be visible in history.
Detailed example: trade salary
If a trade includes salary movement, the rulebook should explain that the salary component is derived from outgoing current-season salary on active or IR players, with taxi excluded, instead of leaving it as vague “trade salary” language.
Detailed example: site navigation
A user wanting to change a roster should not have to understand internal module names. They click the player, see only valid actions, and the site hands them into the correct workspace.
